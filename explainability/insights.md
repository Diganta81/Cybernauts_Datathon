## Comparison with LightGBM gain importance

The gain-based importances from training (fold 1) were topped by
`bal_slope_14d`, `activity_decay_slope`, `activity_momentum`,
`account_age_days`, and `trx_count_90d`, with the weekly `trx_sum_w*` bins
and `channel_shift_ratio` close behind. The SHAP ranking above is led
instead by raw short-window **counts** (`trx_count_7d`,
`trx_count_30d_v2`, `trx_count_30d`, `trx_count_5d`, `trx_count_90d`) with
`days_since_last_trx` close behind, and `bal_slope_14d` /
`weeks_since_active` appearing further down the list.

This is expected and not contradictory: **gain** measures how often and
how effectively a feature is used to *split* across the whole ensemble
(favoring features with many useful thresholds, like continuous slopes
and the 13 weekly bins individually), while **mean |SHAP|** measures the
average *magnitude of impact on the final prediction* per sample. A small
number of count features can each carry a large, consistent
push-toward-churn for the dormant segment of accounts, giving them high
SHAP impact even if any single weekly-bin feature is used in fewer splits
overall. Both views agree on the same underlying mechanism: **recent
transaction volume and recency dominate the prediction**, with
balance-trend features (`bal_slope_14d`) as a secondary but consistent
signal in both rankings.

## ⚠️ Known limitation — model/split mismatch in the SHAP cell

The "EXPLAINABILITY ARTIFACTS" cell runs immediately after the CV training
cell (5-fold, single seed). At that point:

- `lgb_model` is the model object left over from **fold 5** (the last
  iteration of the CV loop), not "fold-1" as the in-code comment states.
- `tr_idx_f1, val_idx_f1 = list(skf.split(X, y))[0]` correctly recovers
  **fold 1's** train/validation split, which is used to compute the
  REGION target encoding (`region_means_f1`) applied to `X_full`.

So `X_full` (and the SHAP sample drawn from it) uses **fold-1's REGION
encoding**, while `explainer = shap.TreeExplainer(lgb_model)` explains
**fold-5's model**. The two folds use the same hyperparameters, the same
137-feature representation, and very similar REGION encodings (each
fold's encoding is a small perturbation of the global churn rate per
region), so this does not materially change the SHAP results — but it is
not the exact "fold-1 model + fold-1 encoding" pairing the comment
describes.

For a future iteration, this should be made explicit by storing a
reference model at the desired fold, e.g. inside the CV loop:

```python
if fold == 1:
    lgb_model_for_shap = lgb_model
```

and using `lgb_model_for_shap` in the explainability cell instead of the
loop's final `lgb_model`.

## Leakage check

- **Temporal cutoff:** all transaction and balance features are computed
  only from records with `TRX_DATETIME` / `DATE` < `CUTOFF`
  (2024-04-01), enforced via a Dask-level filter applied immediately after
  ingestion and before any aggregation. No feature can see data from the
  prediction window (2024-04-01 to 2024-04-30).
- **REGION target encoding:** computed independently within each CV fold
  using only that fold's training-set churn labels (`tr_region_df`,
  `region_means`), then applied to that fold's validation set and to the
  test set. Validation-fold churn labels never contribute to the encoding
  used to score that same fold.
- **No direct label leakage:** `CHURN` is excluded from `FEAT_COLS`
  (`DROP_COLS = ["ACCOUNT_ID", "CHURN", "REGION"]`), and no feature is
  derived from `train_labels` other than through the fold-safe REGION
  encoding above.
- **Account-type scoping:** KYC features are restricted to
  `ACCOUNT_TYPE == "Customer"` (850,000 of the ~1M KYC rows), matching the
  train/test account universe (Merchant and Biller accounts are not
  scored).
- **inf → NaN fix:** the `train_feat.replace([np.inf, -np.inf], np.nan)`
  step (and equivalent for `test_feat`) is applied identically to train
  and test, with imputation medians computed on train only — no
  test-derived statistics leak into train.

## Dependence insight

`trx_count_7d` and `days_since_last_trx` — the two top features by mean
|SHAP| after the count features — show the expected monotonic
relationship with churn risk: SHAP contribution rises sharply as
`trx_count_7d` approaches zero and as `days_since_last_trx` increases past
roughly 2-4 weeks, then plateaus. This is consistent with the dormancy-
driven nature of the underlying churn label — the model has effectively
learned a smoothed "has this account gone quiet recently" detector, with
`bal_slope_14d` and `weeks_since_active` providing secondary corrections
for accounts that look quiet on a 7-30 day view but have a longer-term
balance trend pointing the other way.
